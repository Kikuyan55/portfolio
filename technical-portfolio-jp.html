2. 建築的戦略：役割ベースのアカウント分離 (Confinement)

システム全体の通信において特定の個人アカウントへの依存を排除し、テクニカルリスクを軽減するため、すべてのメールルーティングを役割ごとに隔離されたエントリーポイントへとリファクタリングしました。これにより、厳格なセキュリティ境界と明確なアクセス制御を実現しています。

  • info@bnd-ryugaku.com （サポート・オペレーション窓口）: 学生マネージャーおよびカスタマーサポートチームに割り当て。日々の問い合わせを受信し、すべてのメタデータはn8nを通じてデータベースへと自動的に抽出・記録されます。
  • owner@bnd-ryugaku.com （エグゼクティブ・管理窓口）: 経営判断、B2B（企業間）の重要通信、および重大インシデントへの対応専用として分離。日々の運用ノイズが経営監視レイヤーに影響を与えるのを防止します。

3. コア・アーキテクチャ・パラダイム：ハイブリッド決定論的設計

本設計の基盤となるブレイクスルーは、複雑なマルチ変数ロジック（授業料の算出、滞在形態の組み合わせ、シーズン割引、現地税率の適用など）を処理する際、LLMが抱える確率論的な限界を正確に特定し、解決した点にあります。

💡 技術的インサイト：確率 (Probability) vs 決定論 (Determinism)

どれほど厳格にFew-Shotプロンプトやシステム指示を構築しても、確率論的に動作する言語モデル（Gemini 3.1 Flash-Liteなど）には、多段階の数学的計算においてトークンの揺らぎやハルシネーション（嘘の生成）のリスクが常に伴います。商業運用の世界において、1セントの請求エラーであっても顧客の信頼を失い、法的なリスクを招きます。

解決策： システム全体の「関心の分離」を行いました。LLMは文脈の理解、分類、トーンの調整（確率的なタスク）にのみ使用。一方で、すべての定量的な計算処理は隔離されたJavaScript実行ブロック（決定論的なタスク）へ強制的にバイパスさせ、100%の計算精度を担保する構造に設計しました。

Human-in-the-Loop (HITL) ワークフロー

完全に自律したAIエージェントを本番環境に配置して顧客へ誤情報を送信してしまうリスクを避けるため、効率性を極限まで高めた**Human-in-the-Loop（人間の介入）**モデルを採用。バックエンドシステムが受信メールを処理し、文脈に沿った正確な返信文面と詳細な見積もりテーブルを自動でドラフト（下書き）としてステージングします。これにより、人間側はゼロから文章を作る15分の作業から解放され、10秒の「確認と送信ボタンの押下」のみへと運用負担が凝縮されました。

4. 本番パイプライン実例：顧客インジェクション & データ破壊防止エンジン (Inquiry_Info)

このミッションクリティカルなパイプラインは、新規の問い合わせの受信、文脈分類、ベクトルデータベースに対するセマンティックRAGクエリの実行、およびデータ更新をシームレスに処理します。さらに、データ書き換え時のデータ破損を防ぐ高度なセーフガードを備えています。

エンドツーエンドのデータフロー構造

1. 定期インジェクション層: Gmail API経由で未読のトランザクションインテントをポーリング。
⬇️
2. 文脈 & 言語分類: Gemini 3.1 Flash-Liteがメタデータを抽出し、構造化されたJSONへ変換。
⬇️
3. JavaScript防御パース層: Markdownの不要な記号を剥ぎ取り、オブジェクトトークンをサニタイズ。
⬇️
4. 決定論的マルチブランチルーティング: Switchノードがペイロードの意図に基づいて処理を分岐。
⬇️
5. 同時実行保護: 信頼性の高いGet Row(s)によるデータベースのスナップショット取得。
⬇️
6. セマンティックメモリ検索: Pinecone Vector Index検索により、過去の最適な類似文脈をマッチング。
⬇️
7. 出力生成: 多段階のLLMによる応答オーケストレーション。
⬇️
8. トランザクション完了: データベースへのアトミックな更新後、厳密にMark as Read（既読化）を実行。

n8nの「Undefinedカラム上書き脆弱性」の回避

データの更新処理において、n8nのGoogle Sheets用Update Rowコンポーネントが持つ特有の挙動が判明しました。マッピングの過程でオブジェクトのプロパティが欠落している、あるいはundefinedと評価された場合、コネクタは対象セルに空文字（""）を送信してしまい、過去の履歴データを消去してしまう仕様となっています。

すべての列に対して複雑な条件分岐を手動で組む技術負債を避けるため、**「メモリ・マージ（Memory-Merge）」**デザインパターンを開発。データベースを更新する直前にGet Row(s)ブロックで現在の状態をメモリに読み込み、カスタムJavaScriptブロックを用いてシャローコピーを実行。最新の差分データのみを既存のテンプレートに重ね合わせることで、重要データの損失を防いでいます。 

// メモリ・マージ パターン：iPaaS環境におけるNull上書きの防御
// 入力: item[0] = 新しいメール差分データ, item[1] = 既存のデータベーススナップショット

let currentSheetData = {};
try {
    // データベースにコミットされている現在の包括的な状態を取り込み
    currentSheetData = $input.all()[1].json;
} catch (e) {
    // 過去データが存在しない場合のフォールバック
    currentSheetData = {};
}

// 最新のクライアント通信からパースされた差分パラメータを抽出
let incomingPayload = $input.all()[0].json;

// 既存の履歴カラムを保護しつつ、値を更新するための厳格なシャローマージを実行
let finalPayload = { ...currentSheetData };

for (const key in incomingPayload) {
    if (incomingPayload.hasOwnProperty(key)) {
        let value = incomingPayload[key];
        
        // 明確なアップデートが含まれている属性のみをマージ
        if (value !== undefined && value !== null && value !== "" && value !== "/") {
            finalPayload[key] = value;
        }
    }
}

return { json: finalPayload };

5. 本番パイプライン実例：決定論的財務データ抽出 (Expenses)

運営費や設備投資などの経費処理には、完璧な財務計算が求められます。このパイプラインは、請求書や経費関連の通信を傍受し、トランザクションデータを絶対的な数値精度で処理します。

Regex（正規表現）ファーストのトークンパース戦略

暗黙的なパターンに対してハルシネーション率を0%に抑えるため、インジェクション層では特定の文字列抽出にLLMを使用しません。JavaScript実行ノード内の高速な正規表現エンジンを利用し、一意のトランザクションキー（多部構成の識別子など）を完全に決定論的に抽出した上で、全体的なテキストを**Gemini 3.1 Flash-Lite**へ渡し、セマンティックな文脈分類を行っています。

コードレベルのエグゼキューションコンテキスト

以下は、Inquiry_Infoルーティング内で実装されているコアパースロジックです。トークンのサニタイズ処理、配列の境界防御、および自動フォールバックオブジェクトを備え、LLMから不正な形式のJSON文字列が返された場合でも、パイプラインが絶対に停止しない設計になっています。 

/**
 * 高度なエンタープライズデータ抽出 & サニタイズブロック
 * コンテキスト: LLMによって生成された動的マルチ変数スキーマのパース
 */

// LLMが時折出力するMarkdownのコードブロック枠（```json）を中和・消去する関数
function sanitizeJsonString(rawString) {
    if (!rawString) return "{}";
    let cleanStr = rawString.trim();
    cleanStr = cleanStr.replace(/^```json\s*/i, '');
    cleanStr = cleanStr.replace(/```$/, '');
    return cleanStr.trim();
}

let rawOutput = "";
try {
    // 上流のGemini 3.1 Flash-Liteエージェントから応答トークンをインジェスト
    rawOutput = $input.all()[0].json.choices[0].message.content;
} catch (e) {
    // スキーマ構造に致命的な問題がある場合のフォールバック
    rawOutput = JSON.stringify($input.all()[0].json);
}

let parsedPayload;
try {
    parsedPayload = JSON.parse(sanitizeJsonString(rawOutput));
} catch (error) {
    // パイプラインを稼働させ続けるためのグレースフルなエラーハンドリング
    parsedPayload = {
        error: true,
        parsing_exception: error.message,
        raw_token_fallback: rawOutput,
        timestamp_utc: new Date().toISOString()
    };
}

// 出力フォーマットをn8nの厳格なJSONオブジェクト構造に適合させてリターン
return { json: parsedPayload };

6. 本番パイプライン実例：スマートメーターロギングシステム (Electricity)

この自動化サブシステムは、施設のユーティリティ（電気料金等）を管理するため、物理的な物件パラメーターの推移、計算、およびデルタ（差分）追跡指標を長期的に記録します。

JavaScriptランタイムインバージョンによるn8nのキー上書き回避

この特定の自動化モデルにおいて、Google Sheetsノードをそのまま使用すると、空文字やnullとしてマッピングされた明示的なプロパティが、既存の行のデータを誤って白紙化してしまうという制約に直面しました。

この制限をクリアするため、JavaScript本来のdelete演算子を活用した高度なデータサニタイズ層を実装。割り当てのない静的プロパティをペイロードから完全に「削除」して送信することにより、ノードに対して対象セルへの変更命令そのものを無視させ、意図したカラムのみの部分更新（パッチ適用）に成功しました。 

// ペイロードインバージョン：既存データの変異を防ぐための未変更属性の完全削除
let operationalPayload = { ...$input.all()[0].json };

// 部分更新時に上書きしてはならない静的キーを明示的に指定
const targetedKeysToPreserve = ['historicalId', 'creationTimestamp', 'depreciationBase'];

targetedKeysToPreserve.forEach(key => {
    if (operationalPayload.hasOwnProperty(key)) {
        // キー自体を完全消去することで、下流のコネクタによるセル変更をスキップさせる
        delete operationalPayload[key];
    }
});

return { json: operationalPayload };

データガバナンス & 企業プライバシー

マルチモーダルおよびリッチテキストへの対応

レイアウトが複雑なメールを正確に処理するため、インジェクションワークフロー側でMIMEデータの2段階パース戦略を実行。高精度な意味抽出を行うためにプレーンテキスト（body_plain）を分離してLLMに読ませる一方で、構造化されたHTMLフォーマットも並行して維持し、人間の目による最終確認時に見やすくレスポンシブなテンプレートを出力させています。
